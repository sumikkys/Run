## A. 项目简短介绍

**Run!** 是一个面向 OTA、差旅平台和购票平台的 **误车误机救急插件**。它不做普通地图导航，也不做买票前的常规中转推荐，而是聚焦一个具体、高压、真实的公共交通场景：**用户已经买了票，但因为忘记出发、交通异常、排队、晚点或换乘失败，快赶不上已购车次/航班，此时怎样用最小损失保住行程？**

Run! 的核心能力是：读取或输入用户订单，结合全国铁路时刻表、北京两机场航班快照、高德地图 ETA、北京地铁官方时刻表、北京主要换乘站人工标注微动线、余票票价快照与异常模拟，判断原班次是否还能赶上；如果原站/原航班已经不可达，就搜索 **下游站拦截原车、补救高铁、改签、退票重买、航空兜底、冷门机场 + 火车接驳** 等方案，并输出一张极简行动卡。

一句话：**普通地图问“去哪儿”，Run! 问“你要赶哪张票，现在怎样把损失降到最低”。**

## B. Proposal 摘要

### B.1 项目名称

**Run! 出行救急 - 订单后的误车误机止损插件**

### B.2 赛道归属

**T3 AI × 公共系统**。本项目聚焦公共交通系统中的流程效率、信息匹配、异常检测与服务响应问题。

### B.3 目标用户

- 已在 OTA / 12306 / 航司 / 差旅平台购买火车票或机票的用户；
- 因会议、考试、面试、夏令营、交通异常等原因快赶不上车/飞机的人；
- 差旅平台、OTA、企业行政、机场/车站服务台等 B 端合作方。

### B.4 真问题

用户快赶不上已购票时，最缺的不是路线建议，而是 **30 秒内可执行的止损决策**：

- 还要不要继续冲原站？
- 原票能不能从下游站接回？
- 哪张补救票有余票、能追上、成本最低？
- 改签、退票重买、打车、换飞机哪个总损失最小？
- 如果中途又晚点、没赶上、余票消失，系统如何重新计算？

### B.5 独特生态位

Run! 不是地图 App、不是 12306、不是航班动态工具，也不是 ChatGPT 聊天框。它的位置是：

> **已购订单快失败时的实时止损交易层。**

- 地图 App 主要优化“到达某个 POI”；
- 12306 / OTA 主要优化“买票、改签、中转”；
- 小红书沉淀了真实动线经验，但信息分散；
- 通用 AI 能出建议，但没有订单、余票、支付、叫车和改签闭环。

Run! 把这些能力合成为：**判断原票是否可救 -> 搜索最小损失方案 -> 一键购票/叫车/改签 -> 异常后实时重算**。

### B.6 主 Demo 价值

主 Demo 只做一个强场景：**北京出发火车误车救急**。

用户在北京中关村学院参加活动，已经买了北京西出发去西安方向的火车票。第一次 Run! 提醒时，正常出发还能赶上；用户忙着其他事情忽略。第二次提醒时，继续去北京西已经来不及。Run! 发现原车会经停石家庄，并发现一趟有少量余票的补救高铁可以先到石家庄，于是推荐“买一小段高铁到石家庄，在石家庄接回原车”。

这个 Demo 的爽点是：**普通地图会继续带你去北京西；Run! 会告诉你别去北京西了，去下游站把原车追回来。**

### B.7 AI 的作用

Run! 不是让大模型凭感觉算路线。系统采用 **规则引擎 + 数据层 + DeepSeek API + 行动卡生成** 的架构：

- 规则引擎：负责严肃判断，保证可靠性；
- DeepSeek：负责自然语言解析、复杂结果解释、行动卡生成、自定义车次 Beta 的推理说明；
- 未来探索：AI 自动整合小红书、官方导向图、车站公告，构建城市内微动线知识库。Demo 阶段北京微动线主要由人工标注。

### B.8 商业模式

Run! 最适合嵌入 OTA / 差旅平台订单详情页、出发前提醒页、改签退票页和客服页。

对 OTA 的价值：

- 将一次负面误车体验转化为平台救场能力；
- 提升补救票、改签、打车、保险、会员等转化；
- 减少人工客服压力；
- 提升平台信任和复购。

```text
[ ] 行李状态：轻装 / 大件行李
```

数据使用：

```text
[ ] 全国铁路时刻表：查原车经停站、下游拦截点、补救列车

[ ] 手动标注北京地铁主要换乘站换乘时间
[ ] 北京地铁官方时刻表
[ ] 高德 API：起点到北京各铁路枢纽 ETA
[ ] 余票快照 / 模拟余票
[ ] DeepSeek：解释用户输入、生成行动卡、说明方案风险
```

输出：

```text
[ ] 原站还能赶上：推荐继续赶原站
[ ] 原站赶不上但可追车：推荐下游站拦截
[ ] 原站和下游拦截都不可行：推荐改签/退票重买/航空兜底
```

```text

[ ] 补救车晚点 X 分钟
[ ] 我没赶上补救车

[ ] 地铁换乘失败 / 没赶上
[ ] 余票消失
```

验收标准：

```text
[ ] 输入北京出发真实车次时，系统能返回原车经停表
[ ] 如果原站赶不上，系统能尝试枚举下游拦截点
[ ] 页面清楚标注 Beta，不保证生产级准确
[ ] DeepSeek 生成自然语言行动卡，但严肃判断来自规则引擎
```

### 15.2 为什么不是 12306 / OTA？

```text
12306 和 OTA 主要解决买票、改签、中转查询。Run! 切的是订单后的救急场景：用户已经买了票、已经快赶不上了，系统要把地图、时刻、余票、动线和改签执行合成一个行动卡。
```

### 15.3 为什么不是 ChatGPT？

```text
ChatGPT 可以给建议，但没有订单权限、余票快照、实时 ETA、站内动线和一键执行入口。Run! 嵌在 OTA 订单页，是救急交易闭环，不是聊天顾问。
```

### 15.4 为什么需要 AI？

```text
规则引擎负责算得准，AI 负责看得全、懂得杂、讲得清。DeepSeek 用于理解用户输入、整合复杂结果、生成行动卡，并作为未来自动构建城市微动线知识库的入口。
```

### 15.5 数据不准确怎么办？

```text
Demo 阶段采用真实时刻表快照 + 高德 ETA + 人工微动线 + 余票模拟 + 异常 mock。我们明确标注数据来源和置信度。生产化后可接 OTA、票务、机场、车站等官方接口。
```
